memcpy的优化  

2014-08-21 14:16:55|  分类： 校招beginning |  标签： |举报 |字号大中小 订阅

今天无意中看到了一个写法很geek的memcpy：

void duff_memcpy( char* to, char* from, int count ) {
    size_t n = (count+7)/8;
    switch( count%8 ) {
    case 0: do{ *to++ = *from++;
    case 7:     *to++ = *from++;
    case 6:     *to++ = *from++;
    case 5:     *to++ = *from++;
    case 4:     *to++ = *from++;
    case 3:     *to++ = *from++;
    case 2:     *to++ = *from++;
    case 1:     *to++ = *from++;
            }while(--n>0);
    }
}

开始觉得很无聊。。把语法搞这么奇怪还没什么实际作用。
毕竟像这类函数在库中都从底层进行过优化的，
可以参考去年这几天收藏的memset源码。

后来仔细想了下，觉得有点意思了。
于是又做了另外的4个版本来进行测试。
有几个在count为0的时候逻辑上会出错，
这种细节就没有管了，
重要通过这几个例子记录一下底层优化的一点手段。

一。实验环境：

intel DUO P7350 , VISTA , VC6 ，.C

这里的底层优化是在对底层有所了解的情况下，
使用纯C的技术进行的优化，
不同于使用内联汇编，
这些代码应该能在支持标准C的各种地方编译。

二。对原版duff_memcpy的分析

void duff_memcpy( char* to, char* from, int count ) {
    size_t n = (count+7)/8;
    switch( count%8 ) {
    case 0: do{ *to++ = *from++;
    case 7:     *to++ = *from++;
    case 6:     *to++ = *from++;
    case 5:     *to++ = *from++;
    case 4:     *to++ = *from++;
    case 3:     *to++ = *from++;
    case 2:     *to++ = *from++;
    case 1:     *to++ = *from++;
            }while(--n>0);
    }
}

先不用管那个switch，只管do循环，
那么每一轮，将赋值8个BYTE的数据。
而n是对count按8向上取整的值。
其中的第一次实际上达不到8个，
这个时候就需要switch的跳转作用，
余下的是多少次，就跳到多少的case处。

不过大家也看出来了，当count==0时，
还会至少do一次，实际上是错误的操作。

三。朴素版的memcpy

void memcpy1(char *to , char *from , int count){
    do{
        *to++ = *from ++ ;
    }while ( --count != 0 ) ;
}

为了和那种do结构照应，也没管count为0的情况。

对比前两个版本，
即可想象duff版可能有的作用了。
因为每一轮处理8个数，
那么，除了余数部分外，
是1次比较8次赋值。

而朴素版是1次比较1次赋值。

因为条件跳转对流水线会造成很大的损坏，
所以减少条件跳转，对效率提高很有意义。
不过由于现在CPU都有分支预测，
而对于循环来说，条件会满足N次，
只有最后一次不满足，于是跳出。
只要预测机制够强大的话，
流水线损坏所带来的负面影响可以不用考虑。

尽管如此，多出的7条比较加跳转语句也会有额外开销。

四。利用CPU内在并发性，两路赋值，进行优化

void memcpy2(char *to , char *from , int count){
    char *to0 = to ;
    char *from0 = from ;
    char *to1 = to + 1 ;
    char *from1 = from + 1 ;
    int n = (count + 1) / 2 ;
    
    switch (count & 1){
    case 0: do{ *to1 = *from1 ; to1 += 2 ; from1 += 2 ;
    case 1:     *to0 = *from0 ; to0 += 2 ; from0 += 2 ;
            }while(--n>0);
    }
}

代码的意思实际上是很显然的，
借鉴了一下duff版的写法。

重要区别在于duff版的8个case只能级联执行，
前面to和from执行得到的结果讲影响后面的执行。
而这个2版氛围0和1两个支路，
即使其中一个支路完全暂停，
另外一个支路也可以执行并得到正确的结果。
这就给了CPU发挥其特长的机会，
把这两路无耦合的指令并发执行。

五。玩大一点，8路并发

基本原理与两路并发是一致的，
只是如果一来就放8路的代码，
会把人看晕的，
所以用两路并发做个原理性的铺垫。

void memcpy3(char *to , char *from , int count){
#define DEF(x) char*to##x=to+x;char*from##x=from+x 
#define EXEC(x) *to##x=*from##x;to##x+=8;from##x+=8;
    DEF(0) ;DEF(1) ;DEF(2) ;DEF(3) ;DEF(4) ;DEF(5) ;DEF(6) ;DEF(7) ;
    size_t n = (count+7)/8;
    switch( count%8 ) {
    case 0: do{ EXEC(7) ;
    case 7:     EXEC(6) ;
    case 6:     EXEC(5) ;
    case 5:     EXEC(4) ;
    case 4:     EXEC(3) ;
    case 3:     EXEC(2) ;
    case 2:     EXEC(1) ;
    case 1:     EXEC(0) ;
            }while(--n>0);
    }    
#undef DEF
#undef EXEC
}

其中用到了宏的连接符“##”。
这样只是为了减少书写的代码量。

从后面的测试中是可以看到，
3版比2版更快。
不过并非路数越多越快，
就像不是越多核的CPU越好一样，
核间通信的代价会迅速增长。
所以就不更多路数进行测试了。

该版与duff版比较，效果也是明显的。
同样的循环次数，不同只在于变量间的耦合关系。
3版由于解耦合，比duff版快了不少。

六。利用字长，32比特整体赋值

如果3版能够看懂，这个版本应该就没啥问题。
不一样的仅仅在于，按照32bit进行赋值，
这样可以充分利用CPU的字长，
再次减少循环的数量。

void memcpy4(char *to , char *from , int count){
#define DEF(x) int*to##x=(int*)to+x;int*from##x=(int*)from+x 
#define EXEC(x) *to##x=*from##x;to##x+=8;from##x+=8;
    DEF(0) ;DEF(1) ;DEF(2) ;DEF(3) ;DEF(4) ;DEF(5) ;DEF(6) ;DEF(7) ;
    int rem = count % 32 ;
    int n = (count)/32;
    switch( (count / 4)%8 ) {
    case 0: do{ EXEC(7) ;
    case 7:     EXEC(6) ;
    case 6:     EXEC(5) ;
    case 5:     EXEC(4) ;
    case 4:     EXEC(3) ;
    case 3:     EXEC(2) ;
    case 2:     EXEC(1) ;
    case 1:     EXEC(0) ;
            }while(--n>0);
    }    
    to = (char*)to0 ;
    from = (char*)from0 ;
    while ( rem -- != 0 ) {
        *to ++ = *from ++ ;
    }
#undef DEF
#undef EXEC
}

七。测试函数

5个版本的memcpy都有相同的接口，
可以编个函数来统一测试。

而这里面的宏，自我感觉比较有味道。
第一次在实践中体会了这种精简代码的手段。

如果是以前我可能会：
_judge(memcpy1 , "memcpy1")
这样句调用。
而有宏来中转一次之后，只需要：
judge(memcpy1)
就可以了。

之所以要传进个字符串，
是为了在console上打出函数名，
更方便观察。

typedef void(*FUNC)(char * , char * , int) ;
#define judge(x) _judge(x , #x) 
#define TEST_LEN 577 
void _judge(FUNC f , char *name){
    int i ;
    char from[TEST_LEN] ;
    char to[TEST_LEN] ;
    for ( i = 0 ; i < TEST_LEN ; i ++ ){
        from[i] = rand() % 1000 ;
    }
    printf("judging %s\n" , name) ;
    (*f)(to , from , TEST_LEN) ;
    for ( i = 0 ;i < TEST_LEN ; i ++ ){
        if ( from[i] != to[i] ) break ;
    }
    if ( i != TEST_LEN ){
        printf("wrong!\n") ;
    }else{
        printf("right!\n") ;
    }
}
#undef TEST_LEN

这个测试关注的是正确性方面的。
时间测试的话用clock()弄下就行了，
也可以直接加到这个judge里面来。
